
**Контейнеризация** — метод виртуализации в UNIX, при котором ядро ОС поддерживает несколько изолированных пространств пользователя **без резервирования ресурсов**. Виртуальные машины недоутилизируют ресурсы сервера (overhead costs): процессор, память, диски выделяются на каждую ОС.

Любой контейнер можно остановить, перезапустить или уничтожить, при этом обрабатываемые данные тоже пропадут. Так выработалось правило проектирования приложений: не хранить важные данные в контейнере. Такой подход называют **stateless**.

## Изоляция

### chroot

Системный вызов `chroot` создает для процесса и его дочерних процессов видимость, что корень файловой системы находится в заданной директории на хосте. При этом **относительные файловые пути могут выходить за пределы изолированного дерева каталогов**, приводя к выходу в файловую систему хоста.

### cgroups

В 2006 году компания Google разработала систему разделения ресурсов **control groups** (cgroups), а затем её включили в ядро Linux. Механизм контрольных групп `cgroups` состоит из двух частей: основы (ядра) и контроллеров. 

Контроллеры делятся по назначению:
- CPU
- Memory
- IO
- PID
- cpuset (привязывает процесс к определенному ядру)

### namespaces

Пространства имен, поддерживаемые ядром:
- `user namespace` - реализует разграничение доступа по uid и guid между процессами
- `PID namespace` - позволяет реализовать дополнительный уровень изоляции внутри одного дерева процессов, не позволяя им видеть процессы из других деревьев
- `network namespace` - собственная приватная таблица маршрутизации, наборы IP-адресов, сокеты, fw
- `mount namespace` - позволяет монтировать и размонтировать устройства в одном неймспейсе
- `IPC namespace` - не позволяет отправлять сигналы в другие неймспейсы 
- `UNIX Time-Sharing (UTS)` -  позволяет на одной системе иметь независимые имена хоста и домена для различных процессов

>**💡** Чтобы запустить один процесс в неймспейсе другого процесса, можно воспользоваться утилитой [nsenter](https://man7.org/linux/man-pages/man1/nsenter.1.html). Это пригодится для запуска утилиты curl с хостовой системы в сетевом неймспейсе контейнера, если в контейнере этой утилиты нет и добавить её туда нет возможности


Разделение памяти и ЦПУ между контейнерами осуществляется при помощи cgroups, а между другими ресурсами - при помощи namespaces. Механизм namespaces отвечает за то, чтобы выделяемые контроллерами ресурсы не пересекались между наборами процессов.

Помимо них, Docker использует компоненты Union File System (ФС со слоями), seccomp (ограничивает системные вызовы хоста) и capabilities (выдает права на хосте) 

## Архитектура Docker

Docker Engine это дистрибутив продукта в платной и бесплатной версиях. Он делится на три части:
- Серверная часть
- REST API
- CLI
  
Серверная часть состоит таких компонентов, таких как:

- сontainerd - демон, управляющий жизненным циклом контейнеров
- runC - утилита для создания образов и запуска контейнеров

containerd и runC опираются на спецификацию образов и контейнеров Open Container Initiative (OCI). Также containerd поддерживает Container Runtime Interface (CRI), посредством которого контейнеры интегрируются в оркестратор.

runC дает контейнерам такие функции ядра, как пространства имен и контрольные группы, а также поддерживает средства защиты AppArmor и SELinux для харденинга.
## Dockerfile

`ENTRYPOINT` отличается от `CMD` тем, что при использовании только `CMD` входной точкой является `/bin/sh -c`, и его можно переопределить флагом `--command`. 

>**💡** Для переопределения `ENTRYPOINT` придется пересобирать контейнер.

Для передачи временных файлов внутрь контейнера, например, `requirements.txt`, лучше использовать bind-mounts:
```
RUN --mount=type=bind,source=requirements.txt,target=/tmp/requirements.txt \
    pip install --requirement /tmp/requirements.txt
```

#### Multi-stage builds

Использует несколько инструкций `FROM` для создания контейнера-сборщика и финального контейнера. Полезно в случаях, когда собирается исполняемый файл, который затем используется внутри sketch-контейнера.
