ОУД4 - это уровень доверия, которому должны соответствовать системы Дистанционного Банковского Обслуживания (ДБО) в соответствии с положением 382-П ЦБ РФ.

Этот уровень доверия определяется в стандарте ГОСТ Р ИСО/МЭК 15408-3-2013, являющимся официальным переводом фреймворка ISO/IEC 15408 «Common Criteria for Information Technology Security Evaluation».

Ключевым элементом фреймворка является **объект оценки** (или ОО, например, ДБО, или его компонент), к которому предъявляются **функциональные** **требования к безопасности** (или ФТБ, например, управление доступом или криптография), формулируемые в **задании по безопасности** или **профиле защиты**. ФБО - функциональные возможности безопасности (реализация ФТБ).

То есть, в рамках фреймворка, **разработчик** должен доказать, что он реализовал в ПО определенный набор функций безопасности, что эти функции действительно работают и что нарушитель не может их отключить или обойти. В правдивости этого заявления разработчик пытается убедить **потребителя**, интересы которого представляет независимый **оценщик**. "Убедительность" заявления разработчика определяется "оценочным уровнем доверия" (ОУД) - комплексом мер, которые разработчик должен реализовать в своем жизненном цикле разработки.

В вопросах области применения требования позиция регулятора касается лишь того ПО, которое имеет доступ в интернет - клиентские приложения и фронт-офисное СПО.

## Анализ уязвимостей

Анализ уязвимостей для этого уровня доверия является необходимой мерой безопасности с кодом `AVA_VAN.3`.

Входные данные для анализа:

1. `ADV_ARC.1` - Разработчик должен описать архитектуру подсистемы безопасности своего продукта. В описании архитектуры должно быть продемонстрировано, что функции безопасности его продукта невозможно обойти, т.е. что в нем нет архитектурных уязвимостей.
2. `ADV_FSP.2` - Разработчик должен написать функциональную спецификацию своего продукта.  В ней должен быть **описан весь API**, связанный с реализацией функций безопасности, назначение каждой функции API, способ ее использования, ее параметры, а также описание всех ошибок, сообщения о которых могут появляться в интерфейсе или в логах, причем отдельно выделить ошибки, связанные с результатом некорректного срабатывания функций безопасности.
3. `ADV_TDS.3` - Разработчик должен описать разделение своего продукта на подсистемы, а каждой подсистемы - на отдельные модули. Для каждого модуля, в котором реализуются функции безопасности, должно быть описано, какие функции API он реализует. Для каждой функции безопасности должно быть расписано что-то вроде **диаграммы взаимодействия UML** - через какие модули и через какие функции API этих модулей идут пути исполнения кода при выполнении функций безопасности.
4. `ADV_IMP.1` - Разработчик должен предоставить открытый код системы безопасности, пригодный для развертывания, или тестовый стенд, идентичный продуктовому
5. `AGD_OPE.1` - Разработчик предоставляет руководство по эксплуатации
6. `AGD_PRE.1` - Разработчик предоставляет руководство по развертыванию, исключающее возможность разворачивания ПО в небезопасной среде 

В анализ уязвимостей `AVA_VAN.3` входят следующие требования:
1. Оценщик должен убедиться, что документация полна и ==соответствует исходному коду (= нет НДВ)==.
2. Оценщик должен выполнить поиск информации в общедоступных источниках. На этом этапе проводится поиск ==известных уязвимостей для используемого ПО== и библиотек от третьих лиц
3. Оценщик должен провести анализ документации на предмет потенциальных архитектурных уязвимостей.
4. Оценщик должен провести анализ исходных кодов на предмет потенциальных уязвимостей.
5. Оценщик должен верифицировать все найденные потенциальные уязвимости, проверив их эксплуатабельность. Тестирование проводится из моедли нарушителя, обладающего **Усиленным базовым потенциалом нападения**. Злоумышленник пользуется эксплойтами в открытом доступе и способен дорабатывать их под свои нужды.
6. Если оценщик обнаруживает эксплуатабельные уязвимости, продукт и документация отправляются на доработку и оценка повторяется после устранения уязвимостей.
